Checklist

When you want to start:

In GitHub, go into the /travelagentapp project, and go to Issues. Find an issue that is not assigned (unless you're working with someone on the same issue).

If there is a prerequisite (which means you can’t work on this issue until another is completed), then you’ll see a number in the title.

For example, if you want to tackle an issue with 1.2 in the title, make sure that 1.1 is completed. Or if you want to tackle 2.2, make sure that 2.1 is completed.

Open the issue in GitHub and, on the right, under Assignees, click Assign yourself.

If you're working with someone, put your name on it too in case there are questions about the issue later.

In the issue, click the gear next to Assignees, and click on your name.

In Git Bash, go to the folder where you cloned the project, and type git pull origin develop.

In Git Bash, in the project folder, type git commit -m "your-message-goes here".

Try to remember (especially Daniel) that messages should be present tense: "Add" not "Added."

This doesn't always work, but when we literally step away from our computer and get our head out of the problem, our brain might slow down enough to find the solution.

If not, reach out to the team, or if life is pulling you away, go into the issue on Git Hub and add the label: Having trouble.



When you want to finish:

Before you push your finished product up to the /develop branch, remember that someone else may have been working in the same file and pushed up to /develop since you pulled the files down.

So you need to re-pull from /develop, but wait. If you do that, you'll overwrite what you just did. That's why we're doing this stash command first. It will shelf your changes.

In Git Bash, go to the folder where you cloned the project, and type git pull origin develop to get the latest.

Up above, you stashed your changes, and then re-pulled the files down from Git Hub.

Now you are running this command to apply your changes to the files you pulled down.

Go into Visual Code and open the file that you changed. VS will ask you to approve the changes.

In Git Bash, do a commit and then push to origin develop.

Go to Issues in Git Hub, click on the issue, and click Close Issue at the bottom.

Git Hub Issues

If you find a problem that is outside of the issue you're working on and you can't figure it out, or life is pulling you away from figuring it out.

Create an issue in Git Hub with a simple title and plenty of details inside. Adjust the label to Bug or Enhancement.

A bug would be something that is not working and an enhancement is something that is working, but should be adjusted.

Git Hub Discussions

If you want to post something about the HTML, CSS, or Javascript that everyone needs to know, I set up discussions for each.

In GitHub, go into the /travelagentapp project, and go to Discussions.

Click on HTML, CSS, or Javascript, and put a comment.

Thoughts on Local Branches

Remember that you're commiting changes and then pushing.

  • A commit is committing the changes to a local repo.
  • A push is pushing that commit up to a remote repo (Git Hub).
I could commit everything to my local branch called main, and then specify that I want to git push origin daniel.

Later, when I'm all done, I can specify that I want to git push origin develop.

This is not a good practice because if I forget and just type git push, it will push to main.

A better practice is to work in a local branch that mirrors the remote branch (in my case, daniel). Then it's clear and you won't make that mistake.

Technically, you can't make that mistake because if you just type git push, it won't work and it yells since there's not a default destination for your first name branch.

Sidebar: I actually did set a default to make it easier (I went into my daniel branch and did: git push --set-upstream origin daniel), so I can just type git push.

When I'm all done and ready to push to the branch called develop, I don't want to create a develop branch although I could. I'll just do a git push origin develop.
